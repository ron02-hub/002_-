# ユーザビリティテスト計画書

## 📋 概要

本ドキュメントは、EV走行音アンケートシステムのユーザビリティテストの計画と実施方法を記載しています。

**注意**: 実際のテスト実施に加えて、仮想ペルソナ200名によるシミュレーションテストも実施しました。詳細は[テスト結果分析レポート](./usability-test-analysis-report.md)を参照してください。

---

## 🎯 テストの目的

1. **ユーザビリティの確認**
   - アンケートフローの分かりやすさ
   - 操作の直感性
   - エラーの発生頻度と原因

2. **UXの改善点の特定**
   - ユーザーのストレスポイント
   - 改善が必要な画面・機能
   - 追加すべき機能

3. **アクセシビリティの検証**
   - キーボードナビゲーションの使いやすさ
   - スクリーンリーダーの動作確認
   - 様々なデバイスでの動作確認

---

## 👥 テスト参加者

### 対象者
- **年齢**: 20代〜60代
- **性別**: 男女問わず
- **運転経験**: 有無問わず
- **EV所有経験**: 有無問わず
- **技術リテラシー**: 初級〜中級

### 募集人数
- **目標**: 10-15名
- **最小**: 5名（統計的有意性を確保するため）

### 募集方法
- 社内メンバー
- 知人・友人
- SNSでの募集
- ユーザーテスティングサービス（必要に応じて）

---

## 📝 テストシナリオ

### シナリオ1: 初回利用者のフロー

**目的**: 初めてアンケートに参加するユーザーの体験を確認

**タスク**:
1. ランディングページからアンケートを開始
2. 同意ページで内容を確認し、同意する
3. 属性情報を入力する
4. 音声チェックを完了する
5. 最初の音声評価を完了する
6. 次の音声評価に進む

**観察ポイント**:
- 各ステップでの迷いや躊躇
- エラーの発生
- 操作時間
- 表情や発言

### シナリオ2: 中断・再開機能

**目的**: 中断・再開機能の使いやすさを確認

**タスク**:
1. アンケートを途中まで進める
2. ブラウザを閉じる
3. 再度アクセスして再開する
4. 中断前の状態が復元されていることを確認

**観察ポイント**:
- 再開の手順が分かりやすいか
- 状態が正しく復元されているか
- ユーザーの安心感

### シナリオ3: 音声評価の操作

**目的**: 音声評価画面の操作性を確認

**タスク**:
1. 音声を再生する
2. SD法で評価する（8軸）
3. 購買意欲を評価する
4. 自由記述を入力する
5. 次の音声に進む

**観察ポイント**:
- 音声再生の操作が分かりやすいか
- SD法の評価方法が理解できるか
5. スライダーの操作性
- 自由記述の入力しやすさ

### シナリオ4: エラー時の対応

**目的**: エラー発生時のユーザーの対応を確認

**タスク**:
- 意図的にエラーを発生させる（ネットワーク切断など）
- エラーメッセージの理解度
- リトライの操作

**観察ポイント**:
- エラーメッセージが分かりやすいか
- リトライの手順が明確か
- ユーザーのストレス度

## 📊 評価指標

### 定量的指標

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| タスク完了率 | > 90% | 各タスクの完了/未完了 |
| エラー発生率 | < 10% | エラー発生回数/総操作数 |
| タスク完了時間 | 各タスクの平均時間 | タイマーで測定 |
| クリック数 | 最小限 | 操作ログで記録 |
| 離脱率 | < 5% | 途中で離脱したユーザー数 |

### 定性的指標

- **満足度**: 5段階評価（1: 非常に不満 → 5: 非常に満足）
- **使いやすさ**: SUS（System Usability Scale）スコア
- **学習しやすさ**: 初回と2回目の操作時間の比較
- **エラーからの回復**: エラー発生後の回復時間

## 🔍 テスト実施方法

### 準備

1. **テスト環境の準備**
   - 本番環境またはステージング環境
   - テスト用の音声ファイル
   - 録画・録音機器（許可を得た場合）

2. **テスト資料の準備**
   - テストシナリオ
   - 同意書
   - アンケート用紙（事後アンケート）

3. **観察者の準備**
   - 観察者2-3名
   - 記録係1名
   - ファシリテーター1名

### 実施

1. **事前説明**（5分）
   - テストの目的と流れ
   - 録画・録音の許可取得
   - 質問への回答方法

2. **テスト実施**（30-45分）
   - シナリオに沿ってタスクを実行
   - 思考発話法（Think Aloud）を推奨
   - 観察者は記録を取る

3. **事後アンケート**（10分）
   - SUSスコアの測定
   - 満足度の評価
   - 自由記述でのフィードバック

4. **デブリーフィング**（10分）
   - テストの振り返り
   - 追加の質問

## 📈 結果の分析方法

### 1. 定量的データの分析

- **タスク完了率**: 各タスクの完了率を計算
- **エラー発生率**: エラーの種類と頻度を集計
- **操作時間**: 各タスクの平均時間を算出
- **クリック数**: 効率的な操作ができているか確認

### 2. 定性的データの分析

- **SUSスコア**: 68点以上が目標（平均的なシステムは68点）
- **満足度**: 4.0以上が目標
- **自由記述**: カテゴリ別に分類して分析

### 3. 問題点の優先順位付け

| 優先度 | 基準 |
|--------|------|
| 高 | 複数のユーザーが同じ問題を報告、タスク完了を阻害 |
| 中 | 一部のユーザーが問題を報告、タスク完了は可能 |
| 低 | 軽微な問題、改善の余地がある |

## 📝 付録

### SUSスコア計算式

各質問に対して1（全く同意しない）から5（強く同意する）で評価：

1. このシステムを使いたいと思う
2. このシステムは不必要に複雑だと思う（逆転）
3. このシステムは使いやすいと思う
4. 技術的なサポートが必要だと思う（逆転）
5. このシステムの機能はうまく統合されていると思う
6. このシステムには不整合が多すぎると思う（逆転）
7. ほとんどの人がこのシステムをすぐに使えるようになると思う
8. このシステムは使いにくいと思う（逆転）
9. このシステムを使うことに自信がある
10. このシステムを使う前に多くのことを学ぶ必要があると思う（逆転）

**計算方法**:
- 奇数番号の質問: スコア - 1
- 偶数番号の質問: 5 - スコア
- 全質問の合計 × 2.5 = SUSスコア（0-100点）
